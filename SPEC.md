# 飞书账单自然语言录入 SPEC

## 1. 目标

把用户的一句自然语言账单输入，转成一条可写入飞书多维表格 `明细表` 的记录。

示例：

- 输入：`今天中饭消费 30 元`
- 输出：写入一条账单明细

本期只做 `明细表`，不涉及任何月度统计表。

---

## 2. 范围

### 2.1 本期包含

- 自然语言解析
- 金额、日期、类型、收支类型、支付平台、备注的抽取
- 写入 `明细表`
- 歧义时追问用户
- 重复写入提示确认

### 2.2 本期不包含

- 月度统计表
- 支付宝全量流水同步
- 截图识别
- 历史账单批量清洗
- 自动创建新分类

---

## 3. 目标表

Base：`702的小账单`

目标表：`明细表`

字段：

- `日期`：datetime
- `月份`：select
- `支付平台`：select
- `类型`：select
- `收支类型`：select
- `操作人昵称`：created_by
- `订单号`：text
- `流水说明`：text
- `款项`：number

---

## 4. 输入输出契约

### 4.1 输入

用户的一句中文自然语言，例如：

- `今天中饭消费 30 元`
- `昨晚打车 45`
- `退了外卖 25`

### 4.2 输出

先输出结构化解析结果，再决定是否写入。

建议内部结构：

```json
{
  "action": "create_record | ask_user | reject",
  "records": [
    {
      "date": "2026-06-26",
      "month": "6月",
      "payment_platform": "",
      "category": "餐食（三餐+盒马山姆都算）",
      "income_type": "支出",
      "order_no": "",
      "note": "今天中饭消费",
      "amount": 30
    }
  ],
  "questions": [],
  "reason": ""
}
```

要求：

- `action=create_record` 时才允许写表
- `action=ask_user` 时只能追问，不能写入
- `action=reject` 用于完全无法解析或用户表达无关

---

## 5. 字段映射规则

### 5.1 日期

- 默认按当前时区 `Asia/Shanghai`
- `今天` 解析为当天日期
- `昨天`、`前天` 等解析为相对日期
- 若用户未提日期，默认记为今天
- 需要补录历史账单时，以用户明确说出的日期为准

### 5.2 月份

- 由 `日期` 自动换算
- 格式固定为 `1月` 到 `12月`

### 5.3 收支类型

只允许：

- `支出`
- `收入`

默认规则：

- 消费、付款、买单、退款等普通消费语义，优先记为 `支出`
- 工资、奖金、返现、理财收益等，记为 `收入`
- 退款按负支出处理，不改成收入

### 5.4 支付平台

只使用当前已有选项：

- `支付宝`
- `微信`
- `信用卡`

规则：

- 用户明确提到平台时才填写
- 没提到时留空
- 不强行猜测

### 5.5 类型

`类型` 是最严格的字段。

规则：

- 只能写入当前 `明细表` 里已经存在的枚举
- 不允许新增分类
- 不允许把口语词原样写进去
- 不允许写“近似但不存在”的分类
- 多候选时必须追问，不可脑补

当前确认的枚举值包括：

- `餐食（三餐+盒马山姆都算）`
- `服饰大类`
- `养狗相关费用`
- `车子养护（电费、保养等）`
- `旅行费用（含etc）`
- `停车费及其他交通费`
- `物业费`
- `工资收入`
- `其他收入`
- `兴趣类支出`
- `其他家用`
- `话费`
- `水电费`
- `热植相关`
- `房贷`
- `节日、礼金等`
- `理财收入`
- `小健家招待`
- `医药费`
- `亚琪副业`
- `公司请客`

分类示例：

- `中饭 / 午饭 / 午餐 / 早饭 / 晚饭 / 吃饭 / 外卖 / 盒马 / 山姆 / 餐厅 / 聚餐` -> `餐食（三餐+盒马山姆都算）`
- `打车 / 滴滴 / 出租车 / 地铁 / 公交 / 停车 / 加油 / 高速` -> `停车费及其他交通费`
- `工资 / 发工资 / 奖金 / 年终奖` -> `工资收入`
- `理财 / 基金 / 股票 / 利息 / 分红` -> `理财收入`

如果口语词可能对应多个枚举，必须追问。

### 5.6 流水说明

- 尽量保留用户原始表达的短摘要
- 不要写长篇解释
- 便于后续检索即可

### 5.7 订单号

- 用户提供才写
- 未提供则留空

---

## 6. 解析策略

### 6.1 先规则，后模型

为了省 token，解析优先级如下：

1. 规则抽取金额、日期、平台、明显收支方向
2. 用本地小映射表匹配 `类型`
3. 仅在歧义时追问
4. 仅在规则无法覆盖时，才调用模型做兜底判断

### 6.2 金额抽取

- 支持整数和小数
- 支持带 `元`
- 输入里有多个数字时，只保留最像消费金额的那个
- 若仍有歧义，追问用户

### 6.3 多笔输入

用户一句话包含多笔时，拆成多条记录。

示例：

- `中午吃饭 30，晚上打车 45`

拆为两条独立记录分别处理。

### 6.4 退款

退款统一记为负支出：

- `退了 50 元` -> `支出`，`款项 = -50`
- 类型尽量回到原始消费分类

---

## 7. 追问规则

以下情况必须追问，不得直接写入：

- 金额缺失
- 金额歧义
- 类型无法唯一映射
- 用户表达依赖上下文，但上下文不足
- 用户说“改一下”“补一笔”但无法定位目标记录

追问原则：

- 一次只问最必要的问题
- 给出候选项时，尽量短
- 能让用户用一个词回复最好

示例：

- `这笔记成什么类型？`
- `是今天还是昨天？`
- `这笔是支付宝、微信还是信用卡？`

---

## 8. 写入规则

### 8.1 最小写入字段

建议至少写入：

- `日期`
- `月份`
- `款项`
- `收支类型`
- `类型`
- `流水说明`

`支付平台` 和 `订单号` 只有在明确时写入。

### 8.2 写入前校验

写入前必须检查：

- 日期是否已标准化
- 金额是否为数字
- `类型` 是否在已有枚举中
- `收支类型` 是否为 `支出` 或 `收入`

### 8.3 去重

为避免重复录入，写入前做轻量去重：

- 时间窗口：最近 30 分钟
- 判定条件：`日期 + 款项 + 类型` 完全一致
- 若命中重复，先提示用户是否仍然写入

---

## 9. Token 节约策略

这是实现中的强约束。

### 9.1 原则

- 尽量不用大模型做完整理解
- 尽量用规则和本地映射表
- 输出尽量短，只返回结构化结果
- 不输出推理过程

### 9.2 实现建议

- `类型` 映射表放本地配置，不塞进每次上下文
- 高频表达走确定性规则
- 只有在“多候选”时才让模型参与
- 追问时只给候选，不解释一大段

### 9.3 推荐链路

1. 规则抽取
2. 本地分类映射
3. 必要时追问
4. 确认后写入

---

## 10. 异常处理

### 10.1 可恢复错误

- 用户没给金额
- 用户没给日期
- 分类不唯一
- 平台不明确

处理方式：追问。

### 10.2 不可恢复错误

- Base 无权限
- 字段类型不匹配
- 网络失败
- 飞书 API 返回错误

处理方式：

- 返回明确错误原因
- 告知当前无法写入
- 不要静默失败

---

## 11. 验收标准

以下情况算通过：

1. `今天中饭消费 30 元` 能正确写入 `明细表`
2. `打车 45` 能正确识别或追问
3. `中午吃饭 30，晚上打车 45` 能拆成两条
4. `退了外卖 25` 能记成负支出
5. `请客吃饭 200` 不会擅自编造分类
6. 类型无法唯一映射时，会追问而不是乱写
7. Token 消耗尽量低，普通录入不依赖长推理

---

## 12. 结论

这份 SPEC 的核心约束只有三条：

1. 只写 `明细表`
2. `类型` 只能用已有枚举
3. 尽量用规则实现，省 token
