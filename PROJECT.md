# 飞书账单自然语言录入需求说明

## 1. 背景

当前有一个飞书多维表格用于记录日常账单支出，数据来源主要是支付宝消费。现状是每次都要人工打开表格逐条录入，操作成本高、容易遗漏，也不利于及时记账。

已确认的 Base 信息：

- Base 名称：`<your_base_name>`
- 主表：`明细表`
- Base 地址：`https://my.feishu.cn/base/<your_base_token>`
- 当前用户：`<your_name>`

目标是让用户直接对 agent 说一句自然语言，例如：

> 今天中饭消费 30 元

agent 能自动理解这句话，提取账单信息，并写入飞书多维表格中的账单记录。

---

## 2. 项目目标

### 2.1 核心目标

实现一个可复用的记账能力，支持自然语言输入 -> 结构化解析 -> 写入飞书多维表格。

### 2.2 体验目标

- 用户只需要用一句话描述消费。
- agent 自动补全可推断字段。
- agent 在写入前尽量减少歧义和录入错误。
- 用户可以通过自然语言继续追问或修正，例如“改成午饭”“记到昨天”等。

### 2.3 结果目标

- 支持常见支出录入。
- 支持低摩擦确认。
- 支持稳定写入到指定 Base 的指定表中。
- 支持严格使用现有 `类型` 枚举，不新增、不编造分类。
- 支持尽量节约 token 消耗的实现方案。

---

## 3. 范围定义

### 3.1 本期包含

- 解析自然语言账单输入。
- 提取金额、日期、分类、备注等信息。
- 映射到飞书多维表格字段。
- 写入一条新记录到 `明细表`。
- 对不明确的信息进行追问或给出确认。
- 返回写入结果与可读摘要。

### 3.2 本期不包含

- 自动识别支付宝订单截图。
- 自动同步支付宝全部流水。
- 自动做复杂财务分析报表。
- 自动修改历史账单的批量清洗。
- 自动创建新的 Base 结构。

---

## 4. 使用场景

### 场景 A: 直接记一笔

用户说：

> 今天中饭消费 30 元

agent 解析为：

- 金额：30
- 日期：今天
- 分类：餐饮 / 午饭
- 备注：中饭

然后写入账单表。

### 场景 B: 信息不完整

用户说：

> 30 元

agent 需要追问：

- 这笔消费是什么类型？
- 发生在什么日期？

### 场景 C: 有歧义

用户说：

> 打车 50

agent 可以推断为：

- 分类：交通
- 金额：50

如果表结构要求更细字段，agent 需要补问是否是“日期、备注、出行方式”等。

### 场景 D: 用户修正

用户说：

> 刚才那笔改成 28 元

agent 需要识别上下文中的上一条待写入/已写入记录，并执行修正或重新写入。

---

## 5. 需求拆分

## 5.1 自然语言理解

系统需要从一句话中识别：

- 金额
- 日期/相对日期
- 类别
- 场景
- 商家/地点
- 备注
- 支付方式
- 是否为支出/收入

需要支持中文口语表达，例如：

- “今天中饭消费 30 元”
- “昨晚打车 45”
- “咖啡 18.5”
- “昨天超市买菜 126.3”

## 5.2 数据标准化

需要把自然语言统一成标准字段值，例如：

- “中饭”“午饭”“午餐”统一归类到 `餐饮`
- “打车”“滴滴”统一归类到 `交通`
- “今天”“刚才”“昨晚”转成实际日期时间

这里需要特别强调：`类型` 字段必须严格映射到已有枚举值，不允许 agent 自行生成新分类名。

## 5.3 写入飞书多维表格

系统需要把解析后的结构化数据写入指定 Base 表中。

写入前应确认：

- Base 是否可访问
- 表字段是否匹配
- 必填字段是否齐全
- 枚举字段值是否存在

## 5.4 错误处理

当出现以下情况时，系统要能处理：

- 金额识别失败
- 日期不明确
- 分类无法映射
- Base 权限不足
- 字段类型不匹配
- 网络或接口失败

---

## 6. 交互流程

### 6.1 推荐流程

1. 用户输入自然语言。
2. agent 解析为结构化账单草稿。
3. agent 判断是否完整、是否存在歧义。
4. 如有歧义，先追问用户。
5. 用户确认后，写入飞书多维表格。
6. agent 返回写入成功结果。

### 6.2 可选的低摩擦流程

**触发条件**：当以下条件**全部满足**时，系统可直接写入，不要求用户二次确认：

| 条件 | 说明 |
|---|---|
| 金额 | 已识别到明确数值 |
| 日期 | 已识别为"今天"或明确的相对/绝对日期 |
| 分类 | 在映射表中命中**唯一**已有枚举值 |
| 支付平台 | 已识别或可默认留空 |

如果以上条件**任意一条不满足**，系统必须先向用户追问确认，不允许猜测写入。

具体追问策略：

- **金额缺失** → 追问"多少钱？"
- **金额歧义**（输入中出现多个数字） → 列出候选金额让用户选
- **日期不明确**（无日期且非"今天"） → 追问"哪一天？"
- **分类多候选**（命中多个枚举值） → 列出候选枚举让用户选
- **分类无匹配**（完全无法映射） → 列出全部已有枚举让用户选，或请用户换一种表达
- **支付平台缺失** → 留空，不追问

## 7. 数据模型假设

> 说明：这里已经基于你当前 Base 的真实表结构整理，不再是纯假设。

### 7.1 主表结构

主表：`明细表`

已读取字段如下：

- `日期`：`datetime`
- `月份`：`select`（格式为 `1月`、`2月` ... `12月`，由系统根据 `日期` 自动填写）
- `支付平台`：`select`
- `类型`：`select`
- `收支类型`：`select`
- `操作人昵称`：`created_by`
- `订单号`：`text`
- `流水说明`：`text`
- `款项`：`number`

其中 `操作人昵称` 是系统字段，通常不需要人工写入。

### 7.2 适合自然语言自动填写的字段

- `日期`
- `月份`
- `支付平台`
- `类型`
- `收支类型`
- `订单号`
- `流水说明`
- `款项`

### 7.3 当前字段约束

- `收支类型` 只有 `支出`、`收入`
- `支付平台` 目前有 `支付宝`、`微信`、`信用卡`
- `类型` 是最关键的业务分类字段，当前已有很多枚举值，例如：
  - `餐食（三餐+盒马山姆都算）`
  - `服饰大类`
  - `养狗相关费用`
  - `车子养护（电费、保养等）`
  - `旅行费用（含etc）`
  - `停车费及其他交通费`
  - `物业费`
  - `工资收入`
  - `其他收入`
  - `兴趣类支出`
  - `其他家用`
  - `话费`
  - `水电费`
  - `热植相关`
  - `房贷`
  - `节日、礼金等`
  - `理财收入`
  - `小健家招待`
  - `医药费`
  - `亚琪副业`
  - `公司请客`

#### 7.3.1 类型字段严格映射原则

`类型` 字段必须遵守以下原则：

- 只能写入已有枚举，不允许新增选项。
- 不允许把口语词直接写进 `类型`。
- 不允许创建“近似但不存在”的分类。
- 如果无法稳定映射到已有分类，必须追问用户，不能猜。

示例：

- “中饭”“午饭”“午餐”“早饭”“晚饭”“外卖”“聚餐”“吃饭”“餐厅”“盒马”“山姆” -> `餐食（三餐+盒马山姆都算）`
- 其他词条只有在能明确映射到现有枚举时才写入。

如果一句话对应多个可能枚举，优先追问，不要自动脑补。

## 8. 智能解析规则

### 8.1 金额

- 支持整数和小数。
- 支持“元”字样和省略单位。
- 如果输入中出现多个金额，优先识别最像消费金额的那个，必要时追问。

当前表里的金额字段是 `款项`，类型为 number。

### 8.2 日期

- 支持“今天”“昨天”“前天”“昨晚”“中午”等相对时间。
- 默认使用当前时区的当天日期。
- 如果用户没有说日期，默认记为今天。

当前表里的日期字段是 `日期`，显示格式为 `yyyy/MM/dd`。

3. **无匹配**：口语词完全无法映射 → 列出全部已有枚举让用户选，或请用户用更具体的表达。

### 8.4 备注

系统应尽量保留用户原始表达，作为备注或来源信息，便于日后检索。

当前表里最适合承担这个作用的字段是 `流水说明`。

### 8.5 退款处理

退款统一记为 **负支出**，不记为收入：

- 用户说"退了 50 元" → `收支类型 = 支出`，`款项 = -50`，`类型` 映射到原始消费分类（如能识别）
- 用户说"外卖退款 25" → `收支类型 = 支出`，`款项 = -25`，`类型 = 餐食（三餐+盒马山姆都算）`
- `流水说明` 中保留"退款"关键词，便于后续检索

### 8.6 多笔合并输入

用户一句话中包含多笔消费时，拆分为多条独立记录：

- 用户说"中午吃饭 30，晚上打车 45" → 解析为两条记录分别写入
- 拆分依据：金额出现次数、时间/场景切换词（"中午...晚上...""先...再..."）
- 每条记录独立走决策矩阵判断是否需要追问
- 如果某条信息不完整，只追问不完整的那条，其余正常写入

## 9. 写入策略

### 9.1 直接写入

适用于：

- 解析结果明确
- 字段都能映射
- 风险低

对于当前这张表，比较适合直接写入的最小字段组合是：

- `日期`
- `款项`
- `收支类型`
- `类型`
- `支付平台`
- `流水说明`

如果 `订单号` 有值，也建议一并写入，便于后续去重和追溯。

### 9.2 token 节约策略

为了尽量节约 token 消耗，推荐把实现设计成“短输入、短输出、少轮次”：

- 尽量避免长篇自然语言推理，只做字段抽取和规则映射。
- 优先使用固定模板输出结构化结果，不输出大段解释。
- `类型` 映射只依赖一份短小的本地枚举映射表，不把全部分类说明重复塞进上下文。
- 对于高频场景，走确定性规则，不调用额外的大模型推理。
- 只有在信息缺失或歧义时才追问，能一次写入就不要多轮对话。
- `流水说明` 保留短摘要即可，不要回填冗长原文。
- 如果已经有足够明确的规则，尽量在代码/skill 层完成映射，而不是每次都让模型重新判断。

### 9.3 推荐的低 token 实现方式

建议把整个能力拆成两层：

1. 规则层：处理金额、日期、支付平台、收支类型、固定分类映射。
2. 兜底层：只在规则层无法确定时，才让模型做少量补充判断。

这样做的好处是：

- 高频输入很省 token
- 输出更稳定
- 可控性更高
- 更不容易把 `类型` 写错

### 9.4 先确认再写入

适用于：

- 金额有歧义
- 日期不明确
- 分类不确定
- 命中多个可能记录
- 用户说“改一下”“补一笔”这类上下文依赖表达

### 9.5 幂等性考虑

写入前执行去重检查，规则如下：

- **时间窗口**：检查当前时间前 30 分钟内已写入的记录
- **匹配条件**：`日期` + `款项`（金额）+ `类型` 三者完全一致视为重复
- **重复处理**：提示用户"检测到 30 分钟内已有相同记录，是否仍然写入？"，由用户确认
- **实现方式**：写入前通过飞书 API 查询 `明细表` 最近 30 分钟内的记录进行比对

---

## 10. 非功能需求

- 可靠性：写入失败要有明确错误提示。
- 可解释性：agent 需要告诉用户它解析出了什么。
- 可配置性：分类规则和字段映射要能调整。
- 可维护性：后续能方便新增”收入””退款””转账”等类型。
- 认证处理：飞书 token 过期时，提示用户重新授权，不静默失败。

## 11. 验收标准

以下场景应视为通过：

**正常场景**：

1. 用户输入”今天中饭消费 30 元”，能写入一条正确账单记录。
2. 用户输入”打车 45”，系统能正确识别并写入或追问补全。
3. 用户输入不完整信息时，系统不会胡乱写入错误数据。
4. 系统能把写入结果明确反馈给用户。
5. 系统在字段不匹配或权限不足时，能给出可操作的错误信息。

**异常场景**：

6. 用户输入”中午吃饭 30，晚上打车 45”，系统能拆分为两条独立记录分别处理。
7. 用户输入”退了外卖 25”，系统记为一笔 `-25` 的 `餐食` 支出。
8. 用户输入”请客吃饭 200”，系统识别出 `小健家招待` 和 `公司请客` 两个候选，列出让用户选择。
9. 用户输入”买了个很奇怪的东西 50”，系统无法映射分类，列出已有枚举让用户选择。
10. 30 分钟内重复写入相同金额+类型+日期的记录，系统提示用户确认。

## 13. 推荐实现方向

推荐把它做成一个专用 skill 或一个固定 workflow，核心分三层：

1. 解析层：把自然语言转成结构化账单草稿。
2. 校验层：检查字段、枚举、日期和歧义。
3. 写入层：调用飞书多维表格 API / lark-cli 写入记录。

如果你后续希望更稳定，建议再加一个“映射配置层”，把你的常用口语和表格字段规则抽离出来，后面不用改代码就能调整。

### 13.1 针对当前表的推荐写入策略

建议优先按下面规则落地：

- 用户说“今天中饭消费 30 元”时，自动写入 `明细表`
- `日期` = 今天
- `款项` = 30
- `收支类型` = `支出`
- `类型` = `餐食（三餐+盒马山姆都算）`
- `支付平台` = 如果未提到，则先留空或默认 `支付宝`，这个点需要你确认
- `流水说明` = 原始文本或清洗后的短摘要
- `月份` = 由日期自动映射成对应月份选项
- `订单号` = 若用户明确给出或后续从支付宝流水中同步，则写入

### 13.2 最省 token 的实现建议

如果目标是尽量节约 token，推荐优先采用以下顺序：

1. 先用固定规则做日期、金额、支付平台、收支类型的抽取。
2. 再用一张很小的”口语词 -> `类型` 枚举”映射表做分类。
3. 只有映射失败时，才让模型在候选枚举里二选一或多选一。
4. 输出只保留结构化 JSON，不输出额外分析过程。

这会比”每次都让 agent 完整理解并自由生成分类”更省 token，也更不容易出错。

### 13.3 运行时配置拆分

本需求文档是完整设计参考，但**运行时不应全量塞进上下文**。实际实现时应拆分为两层：

- **运行时上下文**（每次调用加载）：分类映射表、决策矩阵、字段格式约定、日期规则、去重规则
- **完整文档**（按需查阅）：背景说明、场景分析、设计决策、待确认问题

运行时上下文应控制在 150 行以内，其余内容仅在调试、修改规则时查阅。
